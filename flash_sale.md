# System Design Document: 6M Burger Flash Sale

## 1. Problem Statement & Scope
The marketing team is launching a flash sale to give away exactly 6,000,000 free burgers within a 10-minute window. 

**Functional Requirements:**
* Users can click a "Claim" button to get a free burger.
* A user can only claim a maximum of 1 burger.
* The system must never allocate more than 6,000,000 burgers (strict `<=` invariant).
* Only authenticated users are allowed to participate.

**Non-Functional Requirements:**
* **Extremely High Availability:** The system must survive a massive "thundering herd" spike at the exact second the sale begins.
* **Low Latency:** The API must respond in under 100ms.
* **Consistency over Availability (for Inventory):** We cannot risk split-brain over-allocation. 6,000,000 is a hard business limit.
* **No "Stranded" Inventory:** All available burgers must be distributed if there is sufficient demand.

---

## 2. Capacity Planning & Constraints
* **Total Users:** 20,000,000 active app users eligible for the sale.
* **Total Inventory:** 6,000,000 burgers.
* **Duration:** 10 minutes (600 seconds).
* **Average QPS:** 6,000,000 / 600 = 10,000 QPS.
* **Peak QPS (The Thundering Herd):** In flash sales, up to 80% of traffic can hit in the first 10-30 seconds as users obsessively refresh. We must design the API to handle at least **300,000 to 500,000 QPS** at `00:00:00`.
* **Database Write Bottleneck:** Standard relational databases (PostgreSQL/MySQL) will crash and experience massive lock contention if asked to perform 300,000+ ACID transactions per second. We must fundamentally decouple the "Claim" from the "Database Write".

---

## 3. High-Level Architecture

The architecture is divided into three layers: Edge Protection, Fast In-Memory Processing, and Asynchronous Persistence.

1. **Edge/CDN (Cloudflare / AWS CloudFront):** Absorbs static UI traffic, terminates SSL, and uses a Web Application Firewall (WAF) to drop malicious bots.
2. **API Gateway:** Validates JWTs (authentication) and performs basic rate-limiting per user IP so unauthenticated/spam requests never reach the backend.
3. **Claim Service (App Servers):** A fleet of stateless, auto-scaled microservices (e.g., Go or Rust for high concurrency and low memory footprint).
4. **Redis Cluster:** The in-memory source of truth for global inventory and rapid duplicate checking.
5. **Kafka:** An event streaming platform to buffer successful claims.
6. **Order Fulfillment Workers:** Microservices that consume Kafka events at a controlled rate and write them to the database.
7. **PostgreSQL Database:** The persistent storage for finalized orders.

---

## 4. API Design
We need a single, idempotent endpoint to handle the claim.

**`POST /v1/campaigns/free-burger/claims`**

**Headers:**
* `Authorization: Bearer <JWT>`
* `Idempotency-Key: <UUID>` (Generated by the client on page load to prevent double-charging on network retries)
* `X-Forwarded-For: <IP_Address>` (Used for rate limiting)

**Request Body:**
```json
{
  "campaign_id": "free_burger_2026",
  "client_timestamp": "2026-06-17T12:00:01Z"
}
```

**Response Payload (201 Created):**
```json
{
  "status": "success",
  "message": "Burger claimed successfully!",
  "order_id": "ord_987654321"
}
```

**Error Responses:**
* `400 Bad Request`: Missing headers or malformed payload.
* `401 Unauthorized`: Invalid or missing JWT.
* `403 Forbidden`: User has already claimed a burger.
* `410 Gone`: Inventory is completely sold out.
* `429 Too Many Requests`: Client is spamming the endpoint (caught by API Gateway).

## 5. Detailed Data Modeling

### 5.1 Relational Database (PostgreSQL)
We need tables to store the final source of truth.

**Table: `users`** (Assumed to already exist in the wider system)
* `id` (UUID, Primary Key)
* `email` (VARCHAR)
* `account_status` (VARCHAR)

**Table: `claimed_orders`** (The core table for this system)
* `id` (UUID, Primary Key)
* `user_id` (UUID, Foreign Key)
* `campaign_id` (VARCHAR)
* `status` (Enum: `PENDING`, `FULFILLED`, `FAILED`)
* `created_at` (Timestamp, Indexed for time-series querying)
* `idempotency_key` (UUID)

**Database Indexes:**
* **`UNIQUE INDEX idx_unique_user_campaign (user_id, campaign_id)`**: This is the absolute, unbreachable defense against a user getting two burgers. It enforces the rule at the disk level.
* **`UNIQUE INDEX idx_idempotency (idempotency_key)`**: Prevents the same network request from creating two orders if retried.

### 5.2 In-Memory Store (Redis)
Because Redis operates entirely in RAM, it will act as our fast transactional layer to handle the 500k QPS.

* **Global Inventory Counter:**
  * `Key:` `campaign:free_burger:inventory`
  * `Type:` Integer
  * `Initial Value:` `6000000`
* **Idempotency/Duplicate Check:**
  * `Key:` `campaign:free_burger:claimed_users`
  * `Type:` Redis Bloom Filter
  * `Purpose:` O(1) time complexity to check "Has this user ID already claimed?" Uses minimal memory for 20M users.
* **Rate Limiting:**
  * `Key:` `ratelimit:ip:<ip_address>`
  * `Type:` Integer with TTL (e.g., max 5 requests per second per IP).

## 6. Core Mechanics: Handling 500k QPS (Dynamic Chunking)

If 500,000 requests hit Redis concurrently asking for an inventory deduction (`DECR`), Redis will experience severe lock contention, high CPU usage, and network saturation. To solve this, we use **Dynamic Inventory Chunking**.

### The Chunking Mechanism
Instead of App Servers asking Redis for 1 burger every time a user clicks, they ask for a "batch" or "chunk" of burgers.

1. **App Server State:** Each App Server instance maintains an in-memory, thread-safe variable: `local_inventory`.
2. **The Request:** When a user requests a burger, the App Server checks `local_inventory`. If `local_inventory > 0`, it decrements the counter by 1 and instantly approves the user.
3. **Fetching Chunks:** When `local_inventory` drops to 0 (or a low watermark like 20%), a background thread on the App Server makes a single network call to Redis to grab a chunk of 500 burgers.
4. **Atomic Redis Lua Script:** To prevent race conditions between different App Servers, the Redis fetch is executed via an atomic Lua script:

```lua
-- KEYS[1] = 'campaign:free_burger:inventory'
-- ARGV[1] = chunk_size (e.g., 500)
local remaining = tonumber(redis.call('GET', KEYS[1]))
local chunk = tonumber(ARGV[1])

if remaining >= chunk then
    redis.call('DECRBY', KEYS[1], chunk)
    return chunk
elseif remaining > 0 then
    -- Less than a full chunk left; grab whatever is remaining
    redis.call('SET', KEYS[1], 0)
    return remaining
else
    return 0 -- Completely Sold out
end
```

**Why this scales:** If we have 1,000 App Servers, and each requests chunks of 500, Redis only processes 1,000 QPS instead of 500,000 QPS. The actual deduction happens in the App Server's RAM, keeping API latency under 2ms. It also prevents stranded inventory because servers only fetch exactly what they need based on active traffic.

## 7. Abuse Prevention & Exactly-Once Semantics

How do we ensure a user doesn't get two burgers if they bypass the UI and hit the API concurrently with scripts?

**Step 1: The Bloom Filter (Ultra-Fast Pre-Check)**
Before deducting local inventory, the App Server checks the Redis Bloom Filter (`BF.EXISTS campaign:free_burger:claimed_users <user_id>`). 
* If "No", proceed.
* If "Yes", reject the request (`403 Forbidden`). 
* *(Note: Bloom filters have a tiny false-positive rate. If a user is falsely flagged, they are rejected. This is an acceptable business tradeoff—affecting maybe 0.01% of users—to ensure the system survives.)*

**Step 2: Add to Bloom Filter**
If approved, instantly add the user to the filter (`BF.ADD`).

**Step 3: Database Unique Constraint (Absolute Source of Truth)**
If the system proceeds, the claim is approved. The final asynchronous DB write uses an `INSERT ... ON CONFLICT DO NOTHING` relying on the `UNIQUE INDEX (user_id, campaign_id)`. If a duplicate somehow slips past Redis, the database will silently drop the duplicate, ensuring strict compliance.

## 8. Asynchronous Persistence (Kafka to PostgreSQL)

We must not make the user wait for the database to save their order.

1. App Server approves the claim (via local chunking and Bloom filter check).
2. App Server fires an asynchronous JSON message to a Kafka topic: `burger_claims_topic`.
   ```json
   {"user_id": "123", "campaign_id": "free_burger_2026", "timestamp": "..."}
   ```
3. App Server immediately returns `201 Created` to the user. (Requirement met: < 100ms latency).
4. **Kafka Consumers (Workers):** A separate fleet of backend worker services reads messages from Kafka in batches of 1,000 to 5,000.
5. **Bulk DB Write:** The workers perform a Bulk Insert into PostgreSQL. This protects the database; it only has to handle a steady, controlled stream of large batch writes rather than a chaotic flood of individual transactions.

## 9. Failure Scenarios & Disaster Recovery

* **App Server Crashes while holding local inventory:**
  * *Impact:* If a server pulls 500 burgers from Redis and loses power instantly, those 500 burgers are "stranded" in dead memory.
  * *Mitigation:* This is an acceptable tradeoff. In a 6,000,000 campaign, losing a few chunks (e.g., 1,500 burgers) is a 0.025% margin of error. Prioritize system survival over mathematical perfection.
* **Redis Primary Node Fails:**
  * *Impact:* Global inventory tracking halts.
  * *Mitigation:* Run Redis Cluster in a Multi-AZ topology. If the primary dies, Sentinel/Cluster-manager promotes a replica in milliseconds. We use `WAIT` for synchronous replication of the main inventory key to avoid data loss on failover.
* **Kafka Goes Down:**
  * *Impact:* Approved claims cannot be forwarded to the DB workers.
  * *Mitigation:* App servers detect Kafka unavailability and seamlessly fallback to writing the claims to local disk logs (`/var/log/claims_fallback.log`). A sidecar daemon (like Fluentbit) watches this file and forwards the data directly to the DB or back to Kafka once systems are restored.
* **PostgreSQL Goes Down:**
  * *Impact:* Final persistence is halted.
  * *Mitigation:* Kafka retains messages for 7 days. Once PostgreSQL is brought back online, the workers simply resume consuming from their last committed Kafka offset. No data is lost.

## 10. Observability & Operational Readiness

During the 10-minute window, the engineering team relies on a live dashboard.

* **Key Metrics (Prometheus/Grafana):**
  1. `api.latency`: P99 must stay < 100ms.
  2. `redis.commands_processed_per_sec`: Ensure Redis is not becoming a bottleneck.
  3. `kafka.consumer_lag`: If this spikes rapidly, our DB workers are too slow (autoscale them).
  4. `inventory.global_remaining`: Tracked via the Redis global count.
  5. `inventory.local_exhaustion_rate`: How often App Servers are requesting new chunks.
* **Kill Switch:** A manual configuration flag in AWS AppConfig or LaunchDarkly to instantly disable the `/claims` endpoint (returning `410 Gone`) if the infrastructure begins to melt, thereby protecting the rest of the company's core services.
